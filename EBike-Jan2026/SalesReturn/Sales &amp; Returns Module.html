Sales & Returns Module Requirements (Client Narrative)

Explained by Sam, Front Counter Sales

Introduction

Hi, I am Sam. I work at the front counter — I help customers find what they need, process their payments, and handle returns when something comes back. I do not design software, but I know exactly how a sales system needs to behave so I can do my job without errors.

The purpose of this document is to explain what the system must accomplish, not how to build it. I will describe what I expect as the end user — the client — across the two major parts of the process:

  • The Sales workflow (customer search, product selection, cart, and checkout)
  • The Returns workflow (invoice lookup, return entry, and refund)

This is the same information I would give to a software team building this system for me.

1. How I Work at the Counter

I need to move freely between the steps of a sale — searching for a customer, picking products, reviewing the cart, and checking out. The system must let me jump between these steps without losing my work. If the customer wants to add another item after viewing the cart, or if I need to go back and change something, everything stays in memory until I explicitly cancel.

If I decide to cancel entirely, the system clears everything — customer, cart, coupon — and takes me back to the beginning. But it must warn me first so I do not accidentally lose a sale I have been building.

2. Customer Search

Every sale and every return starts the same way: we must identify the customer. The system cannot let me move forward without choosing the right person.

I search for customers by partial phone number. I type in part of a phone number, and the system shows me anyone who matches. From there, I pick the person I am dealing with.

I need to see:

  • Customer name
  • Address
  • City
  • Phone number

A Clear button resets the search field so I can see all customers again or start a new search.

Once a customer is selected, the sale can begin. If I try to continue without selecting someone, the system must stop me.

If I have already started building a sale and then select a different customer, the system must warn me first — because switching customers will clear the entire cart and any coupon I have applied. That is a safeguard. I do not want to accidentally lose an order I have been building just because I selected the wrong name by mistake.

3. Category & Item Selection

Once I choose a customer, I move into the screen where I pick what they want to buy. The first thing I select is a category.

3.1 Browsing Categories

The category list must start with a special category called All. This shows every part in the system and must appear as the first option. The count beside it shows the total number of parts available.

Every category must show a count of the number of distinct parts in it — not QOH, just how many items belong there.

Examples:

  • All (112)
  • Apparel (7)
  • Lighting (3)
  • Brakes (12)

3.2 Selecting a Part

The item list does not show anything until I have picked a category first. No category, no parts — this prevents me from scrolling through hundreds of items without context.

I need to see:

  • Description
  • Selling Price

I should not see the QOH here — but the system must still enforce it behind the scenes. If QOH is 0, the item must not be addable.

When I find the item the customer wants, I enter the quantity they need. The system will not let me enter less than 1 or more than what is in stock. Then I click Add to put it in the cart. If I change my mind about the selection, a Clear button resets the category, part, and quantity fields so I can start fresh.

There are a few rules the system must enforce when I am adding items:

  • If I try to add an item that is already in the cart, the system must increase the quantity on the existing row instead of adding a duplicate line.
  • If an item is discontinued, it must not be addable.
  • “Add to Cart” must be disabled if selling price is missing.

4. Cart Screen

After I add items, I move to the cart. This is where the customer decides quantities and makes final choices. This is also where mistakes tend to happen if the system is not strict enough.

4.1 What I Need to See in the Cart

  • Part description
  • Quantity on Hand
  • Price
  • Quantity
  • Line total (qty × price)
  • Subtotal, tax, and total

4.2 How I Work With the Cart

  • Quantities cannot be negative.
  • Quantities cannot be zero. If a quantity is set to zero, the item should be removed from the cart.
  • I cannot increase quantity beyond QOH. The system must prevent it every single time.
  • If I add an item that is already in the cart, the system must increase the quantity on the existing row — no duplicate lines allowed.

Totals update immediately when quantities change. When everything looks right, I go to Checkout.

5. Checkout

This is where the full summary is shown, and the sale is completed. Nothing should be left unclear at this point.

5.1 What I See at Checkout

  • Subtotal
  • Discount (only shown if a coupon has been verified — otherwise it does not appear)
  • GST (calculated after the discount)
  • Total
  • Payment Type (required) — the customer picks how they are paying: Cash, Debit, or Credit. One of these must be selected before the sale can go through.

5.2 What Happens When a Customer Has a Coupon

A coupon always uses DiscountPercent. No dollar discounts. The system must apply the percentage discount before tax.

If the customer has a coupon, I type the code into the coupon field and click Verify. The system checks whether the coupon is valid. A valid coupon means today falls within its start and end dates (inclusive), and the coupon has not been removed from the system. If the coupon is valid, a checkmark appears beside the code and the discount applies automatically to the subtotal. If I need to remove it, I click Clear Coupon and the discount goes away.

If I try to complete a sale with a coupon code typed in but not verified, the system must warn me. I should not accidentally miss applying a customer’s discount, but I also cannot apply an unverified one.

There are a few additional rules around coupons:

  • Only one coupon may be applied per sale.
  • Discount must be recorded on the invoice.
  • Tax must be recalculated after the discount is applied.

5.3 What Must Be Ready Before Completing a Sale

Before I can complete a sale, three things must be true: I have selected a customer, there is at least one item in the cart, and I have chosen a payment type. If any of those are missing, the system must block the sale — the Complete Sale button stays disabled until everything is ready.

5.4 What the System Must Do When I Complete a Sale

When the customer pays, the system must:

  • Create an Invoice
  • Create InvoiceLines for each item
  • Store the payment type
  • Store the coupon information (if any)
  • Record the employee completing the sale
  • Set InvoiceDate = today

5.5 What Happens to Inventory After a Sale

  • For each InvoiceLine, QOH decreases by the quantity sold.
  • The system must prevent QOH from going negative.

5.6 After Saving the Sale

Once the sale goes through, I need clear confirmation that it worked — no ambiguity:

The system must clearly tell me the sale is done and show me the Sale ID so I have a reference number. Once I acknowledge the confirmation:

  • The cart must clear completely.
  • The screen returns to the Customer Search page.
  • I cannot continue editing a completed sale.

5.7 Cancelling a Sale

Sometimes a customer changes their mind entirely. If I click Cancel, the system clears everything — the customer, the cart, the coupon, all of it — and takes me back to the Customer Search screen. But before it does, it must warn me that I am about to lose everything I have entered. I do not want to accidentally cancel a sale I have spent time building.

6. Returns

Returns need to be handled carefully. The system must protect inventory, protect customer trust, and make sure we never return more than we sold.

6.1 Looking Up the Sale

Every return begins with entering the Invoice ID. The system loads the exact sale.

If I have already started entering return quantities and then search for a different invoice, the system must warn me first — because everything I have entered will be cleared. It is the same principle as switching customers during a sale: never lose my work without asking.

For each item, I must see:

  • Quantity purchased
  • Quantity previously returned
  • Original selling price
  • Quantity to be returned
  • Reason for the return

The “previously returned” number is not just from one return — it is the total across every return ever processed against this invoice. If the customer returned 2 helmets last month and 1 last week, that column shows 3. The system has to look at all past return records for this invoice to calculate that correctly.

And most importantly:

  • If the part is non-refundable, it cannot be returned.

The system makes this completely clear: if a part is non-refundable, or if the customer has already returned everything they bought of that item (original quantity minus previously returned equals zero), the return quantity field is grayed out and disabled. I cannot even type in it. There is no room for error.

6.2 Entering the Return

The system must prevent:

  • Returning more than purchased
  • Returning more than (purchased – previously returned)
  • Returning non-refundable items
  • Returns without a reason

6.3 How the Refund Works

The refund must honor the original pricing and discount. Here is exactly how the refund math works:

  • Subtotal = sum of (return qty × original price) for each returned item
  • Discount = Subtotal × the original sale’s discount percentage — because we honor the discount the customer originally received, even if the coupon has since expired
  • Tax = 5% of (Subtotal − Discount)
  • Total Refund = Subtotal − Discount + Tax

The system must calculate all of this automatically and update in real time as I change return quantities.

6.4 What the System Must Do When I Complete a Return

Before the system actually processes anything, it asks me to confirm — because a completed return cannot be reversed. I need that moment to double-check the quantities and reasons one last time.

When I complete the return, the system must:

  • Create a Return record
  • Create ReturnedLine records
  • Record reason for return
  • Record EmployeeID

Inventory must be adjusted:

  • QOH increases for all returned items — non-refundable items cannot be returned in the first place, so anything that makes it through goes back into stock.

The refund goes back the same way it came in — if the customer paid cash, the refund is cash. The system knows the original payment method from the sale record.

6.5 After the Return Is Completed

The system must clearly tell me the return is done and show me the Return ID so I have a reference number. Once I acknowledge the confirmation:

  • The return screen resets
  • I return to Customer Search
  • I cannot edit a completed return

6.6 Cancelling a Return

If I decide not to process a return at all, a Cancel button clears everything — the invoice search, the return quantities, the reasons — and resets the screen completely. The system must warn me before clearing if I have already started entering return data.

7. What the System Must Never Do

To be absolutely clear, there are certain things this system should never do:

  • Never assume the customer wants to return everything.
  • Never assume tax or discount is the same for all items.
  • Never allow returning non-refundable items.
  • Never allow QOH to go negative.
  • Never apply multiple coupons.
  • Never continue editing after a sale or return is completed.
  • Never switch customers or invoices without warning me first.
  • Never let me complete a sale without a customer, a cart item, and a payment type.

8. Summary of Expectations

Customer Search

  • Search by partial phone number
  • Selecting a customer is required
  • Switching customers clears the cart and coupon — must warn first

Navigation

  • Buttons always visible: Customer Search, Product Selector, View Cart, Checkout, Cancel
  • Cancel resets everything and returns to Customer Search — must warn first

Product Selection

  • Category list must begin with All
  • Each category shows a part count
  • Parts list is empty until a category is selected
  • Cannot add discontinued or zero-QOH items
  • Must enter quantity before adding (≥ 1, ≤ QOH)
  • Duplicates increase existing row quantity

Shopping Cart

  • QOH must always be enforced
  • No negative or zero quantities
  • Totals update instantly

Checkout

  • Payment type required (Cash, Debit, or Credit)
  • Coupon uses DiscountPercent
  • Coupon must be verified before applying — warn if unverified at checkout
  • Discount applies before tax
  • One coupon per sale
  • Sale requires: customer selected, item in cart, payment type chosen
  • Invoice created with proper records
  • Sale ID shown on completion

Returns

  • Non-refundable items cannot be returned (field disabled)
  • Fully returned items cannot be returned again (field disabled)
  • Must show previously returned quantities (total across all prior returns)
  • Cannot exceed purchased quantity
  • Refund must match original pricing & discount (even if coupon expired)
  • QOH increases for all returned items
  • Confirmation required before processing
  • Searching for a new invoice clears entered data — must warn first
  • Refund method matches original payment method

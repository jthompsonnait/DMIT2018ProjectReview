Customer Selection — Decision Flow

  User selects a customer
       │
       ├── Any service data already on screen?
       │         │
       │        NO  ──→ Load vehicles, populate dropdown
       │
       └── YES ──→ Warn: switching clears services, parts, coupon
                    │
                    ├── Confirmed ──→ Clear all, load new customer's vehicles
                    └── Cancelled ──→ Keep current customer, no change

Screen 1 is only for choosing the customer and vehicle. Once both are selected, the user moves on to Screen 2 for service entry.

Standard vs Custom Service — Decision Flow

  User is adding a service entry
       │
       ├── Standard service selected from dropdown?
       │         │
       │        YES ──→ Description auto-fills (read-only)
       │                Hours auto-fills (editable — user can override)
       │
       ├── Custom text entered?
       │         │
       │        YES ──→ Description = custom text (editable)
       │                Hours = empty (user must enter)
       │                Overrides any standard selection
       │
       └── Neither selected?
                 │
                 └──→ Add Service button stays disabled

Custom always wins. If both a standard service is selected and custom text is entered, the custom text overrides.

Register Requirements

Register is enabled when ALL of these are true: - A customer and vehicle are selected - At least one service entry in the service cart - Parts are optional — a job can have services only

What Register does: Saves job record + all service entries + all parts, decrements QOH by quantity used for each part, assigns and displays a Job ID, locks the entire form (all inputs disabled, Register and Cancel buttons disabled).

D2 Security: Coupon area is restricted to Parts Manager role only (Mechanic blocked).

How Services Move

  ┌───────────────────┐              ┌──────────────────┐
  │  ADD SERVICE FORM │ ──Add──────> │  SERVICE CART     │
  │  (entry fields)   │              │  (running list)   │
  └───────────────────┘              └──────────────────┘
                                            │
                                       ❌ Delete
                                            │
                                            ▼
                                        Removed

  Add:    Entry moves from form → cart. Fields clear for next entry.
  Delete: Entry removed from cart. No confirmation needed (local data).

  Every add or delete recalculates Labour Subtotal immediately.

How Parts Move

  ┌───────────────────┐              ┌──────────────────┐
  │  AVAILABLE PARTS  │ ──+Add────> │  PARTS CART       │
  │  (by category)    │              │  (selected parts) │
  └───────────────────┘              └──────────────────┘
                                            │
                                       ❌ Delete
                                            │
                                            ▼
                                        Removed

  +Add:   Part enters the cart with entered quantity.
          If part already in cart → increase quantity (no duplicate lines).
          Combined quantity must still ≤ QOH.
  Delete: Part removed from cart entirely.

  Parts appear in BOTH grids simultaneously — the available list always
  shows stock, and the cart shows what's been selected for this job.

  Every add or delete recalculates Parts Subtotal immediately.

Coupon Verification Flow

  User enters coupon code and clicks Verify
       │
       ├── Code exists in system?
       │         │
       │        NO  ──→ ✗ displayed, no discount applied
       │
       └── YES ──→ Check date range (StartDate ≤ today ≤ EndDate)?
                    │
                    ├── NO  ──→ ✗ displayed, no discount applied
                    │
                    └── YES ──→ Not RemoveFromView?
                                 │
                                 ├── Removed ──→ ✗ displayed, no discount applied
                                 │
                                 └── Active ──→ ✓ displayed
                                               Discount % applied to Subtotal
                                               Totals recalculate immediately

Coupon must be percentage-based, date-validated, not removed from view, and applied before tax. Only one coupon per job. Accepts any valid coupon (not restricted to service-specific).

Calculation Formulas

Extended Price  = Hours × $65.50                       (per service entry)
Labour Subtotal = SUM(Extended Price)                  (all service entries)

Part Total      = Selling Price × Quantity             (per part in cart)
Parts Subtotal  = SUM(Part Total)                      (all parts in cart)

Subtotal        = Labour Subtotal + Parts Subtotal
Discount        = Subtotal × CouponPercent             (only if coupon verified)
GST             = (Subtotal − Discount) × 0.05
Total           = Subtotal − Discount + GST

Worked Example

Services:
  Oil Change:      0.50 hrs × $65.50 =  $32.75
  Fluid Flush:     1.00 hrs × $65.50 =  $65.50
  Detail Vehicle:  2.00 hrs × $65.50 = $131.00
                                       ────────
Labour Subtotal:                        $229.25

Parts:
  Front Tire:      1 × $125.00       = $125.00
                                       ────────
Parts Subtotal:                        $125.00

Subtotal  = $229.25 + $125.00         = $354.25
Discount  = $354.25 × 0.10            =  $35.43
GST       = ($354.25 − $35.43) × 0.05 = $318.82 × 0.05 = $15.94
Total     = $354.25 − $35.43 + $15.94 = $334.76

What the System Must Never Do

  • Never allow using parts when QOH is insufficient
  • Never save incomplete services or parts (missing description, zero hours, zero qty)
  • Never register a job without at least one service entry in the service cart
  • Never allow editing once the job is registered (write-once only)
  • Never hard-delete job history — use RemoveFromViewFlag only
  • Never skip required validation
  • Never apply a coupon without successful verification
  • Never allow the same part twice in the cart (duplicates must increase quantity instead)
  • Never allow QOH to go negative

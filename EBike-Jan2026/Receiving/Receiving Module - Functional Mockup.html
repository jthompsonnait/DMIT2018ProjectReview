How Unordered Items Work

Items that arrive but are NOT on the PO get documented for return to the vendor.

  • User fills in Description, Vendor Part #, and Quantity, then presses Insert
  • Item appears in the unordered items grid; input fields clear for next entry
  • All three fields are required before Insert is enabled
  • Reason is auto-set to “Incorrect item” — not a user choice
  • Delete removes a row from the grid (local only, no confirmation needed)
  • Clear resets the input fields without affecting the grid
  • Unordered items are NOT saved until Receive or Force Close is pressed

Button State Matrix

What each button does depends on the current state of the receiving session:

Button PO loaded, no changes Data entered Force Close reason filled
Receive Disabled (nothing to process) Enabled — processes all data Enabled — processes all data
Force Close Disabled (no reason) Disabled (no reason) Enabled — processes data THEN closes PO
Reset Disabled (nothing to clear) Enabled — warns first, then clears Enabled — warns first, then clears

Receive Requirements

Receive is enabled when ANY of these are true: - At least one line has Received > 0 - At least one line has Returned > 0 - At least one unordered item has been inserted

What Receive does: Validates all data, records quantities, updates inventory (QOH up, QOO down), auto-closes the PO if all outstanding quantities reach zero.

Force Close Requirements

Force Close is enabled when ALL of these are true: - The Reason for Closing text field is filled in - User has Shop Manager role (D2 security — Store Staff blocked)

What Force Close does: Processes current session data exactly like Receive, then removes remaining outstanding QOO for each part on this PO (this vendor only) and closes the PO.

Reset Requirements

  • Must warn: “You have unsaved changes. Are you sure you want to reset?”
  • Clears: all Received quantities, all Returned quantities, all Reasons, all unordered items
  • Does NOT close or navigate away — stays on the same PO

Valid Return Reasons

Reason When Used
Overship Vendor sent more than what was outstanding
Damaged on arrival Product is not in acceptable condition

“Incorrect item” is reserved for unordered items only (auto-set by system). “Refused by receiver” is NOT a valid reason — never use it.

Force Close QOO Cleanup

For EACH part on this PO:
  QOO decrease = Ordered - TotalPreviouslyReceived - ReceivedToday

  This removes only the unreceived remainder from QOO.

Force Close Example

Part 109 — "Air Filter"
  Ordered from Vendor A:      75
  Previously received:        42
  Received today (this session): 0
  ──────────────────────────────
  QOO decrease for Part 109:  75 - 42 - 0 = 33

  Same part also on order from Vendor B (182 units):
  Vendor B's QOO stays at 182 — untouched.

Force Close requires user confirmation: “Are you sure you want to force close this PO? This action cannot be undone.”

What the System Must Never Do

  • Never auto-fill receiving quantities — user types what actually arrived
  • Never allow Received > Outstanding for any line
  • Never add returned or damaged items into inventory (QOH)
  • Never close a PO unless all quantities are received OR an explicit force close with reason
  • Never adjust another vendor’s QOO when force closing this vendor’s PO
  • Never overwrite previous receiving records — each session is its own entry
  • Never allow a force close without a reason AND confirmation
  • Never allow unordered items to enter inventory or modify the PO
  • Never allow “Refused by receiver” as a return reason
  • Never process an empty submission (no received, no returned, no unordered)
  • Never hide real history using RemoveFromViewFlag
  • Never assume the shipment matches the PO — the user decides what arrived

Receiving Module Requirements (Client Narrative)

Explained by Emily, Receiving Clerk

Introduction

My name is Emily. I started here as a shelf stocker, moved into operating the forklift, and for the last eight years I have been working in Receiving. I do not know anything about programming, but I use this system every day. I deal with deliveries, packing slips, shortages, overshipments, damaged goods, and keeping our inventory accurate.

The purpose of this document is to explain what the system must accomplish, not how to build it. I will describe what I expect as the end user — the client — across the two major parts of the process:

  • The Purchase Order Lookup step
  • The Receiving and Processing step

This is the same information I would give to a software team building this system for me.

1. Starting My Day in Receiving

My day really starts when a truck arrives at our loading dock and a shipment comes in. Before I start entering data, I look at the paperwork and match the shipment to the right Purchase Order (PO).

Because we are a smaller company and we are careful with cash flow, we follow one simple rule that prevents mistakes:

We allow only ONE open (placed but unclosed) Purchase Order per vendor.

In practice, that means:

  • A vendor never has two open POs at the same time
  • If a vendor has no open PO, that vendor should not appear in the receiving list
  • If a vendor does appear, I know exactly which PO matches the shipment I am looking at

That rule prevents me from having to figure out which PO matches a shipment — if a vendor appears, I know exactly which order it is.

Whenever I start a receiving session, the system has to record that I am the one doing the work (my Employee ID) and that it is happening today (Receive Date). If someone asks later, the system should have the answer.

1.1 The First Thing the System Must Allow

A. Look up an existing Purchase Order

When I go into the receiving screen, I do not want a list of every order that was ever created. I only care about the orders that are ready to be received.

The system should show only:

  • POs that have been placed (an Order Date has been set), and
  • POs that have not been closed yet (Closed = false)

No drafts. No closed orders. Just the active order waiting to be received. This way, I am never accidentally working on the wrong order.

The list should show me the following information to identify the right order:

  • PO Number — so I can match it to the paperwork
  • Order Date — so I know how old the order is
  • Vendor — the company name
  • Contact Number — in case I need to call the vendor about the shipment
  • A View Order button — so I can open that PO and start receiving

That is all I need. The list should be clear, simple, and focused on what matters.

B. Select the PO I am about to receive

Once I choose a PO, that becomes my workspace for this receiving session. From my point of view, I am now working with that order, and everything I enter will affect that PO.

2. What I Need to See When I Select an Order

When I open a PO, I need to be able to understand it quickly. That means I need a combination of header information (about the vendor and the order itself) and line details (what was actually ordered).

For the PO header, I need:

Order Information

  • Vendor name
  • PO number
  • Date the order was placed
  • Vendor phone number — in case I need to call the vendor right away about this shipment

For Each Item on the Order

For each item on the order, I need to see all of this in one row so I can work through the line quickly:

  • Part ID — so I can match it to labels on the shelf and the packing slip
  • Description — the item name, so I know what I am looking at
  • Quantity Ordered — how many we originally asked the vendor to send
  • Outstanding — how many are still left to receive. The formula is: Ordered minus everything we have already accepted in past sessions
  • Received — this is where I type in how many good items I am accepting today
  • Returned — this is where I type in how many I am sending back (damaged, overshipped, and similar situations)
  • Reason — if I am returning anything on that line, I have to say why

That combination tells me the complete picture in one view: what we expected, what is still waiting, and what is happening today.

The Outstanding number is important because it tells me what is left. The formula is: what we ordered, minus everything that has been accepted in all previous receiving sessions for this order. If we ordered 50 and past sessions accepted 35, then today I know I am expecting up to 15 more.

3. How I Actually Receive Items

Here is the reality: the shipment almost never matches the paperwork perfectly. That is why my role exists. I never assume the boxes match the PO or the packing slip. I count and inspect everything myself.

The system must allow me to enter:

The quantity received today — for each item on the PO.

Real-world receiving is not predictable. On any day I might deal with:

  • Partial shipments (we got some items, not all)
  • Short shipments (we got less than the vendor said they sent)
  • Damaged goods
  • Overshipments (the vendor sent more than we ordered)
  • Items we did not order at all
  • Backorders (nothing arrived for that line today)

The Receiving Module has to let me record what actually arrived in the building, not what the paperwork says should have arrived. The system should follow my reality, not the other way around.

4. Receiving Rules the System Must Follow

To keep our inventory trustworthy, there are some rules the system has to follow every time I receive items. These rules are non-negotiable from an operations point of view.

Rule 1: I can never receive more than the outstanding quantity

If we ordered 10 and already received 7:

  • Outstanding = 3
  • Maximum receivable today = 3

If the vendor sends more than 3, I handle it this way: I enter what I can accept in the Received column — the system will not let me set this higher than the outstanding amount — and then I enter the extras in the Returned column with “Overship” as the reason. The system does not split anything for me automatically; it just makes sure I cannot accidentally receive more than we are owed.

Overshipped items are common. If the system let me receive all of them, our inventory numbers would become incorrect. That is why this rule is so important.

Rule 1A: A Return Reason is required for all returned quantities

Any time I send items back — even a single unit — I need to say why. The system must force me to choose a Return Reason.

Valid reasons include:

  • Overship — the vendor sent more than what was outstanding
  • Incorrect item — this reason only applies to unordered items
  • Damaged on arrival — the product is not acceptable as received

We do not use vague reasons like “refused by receiver.” Returns happen because of the vendor’s mistake or the product’s condition, not for unspecified reasons.

Returned quantities must never go into inventory. The system should track returned items for history and follow-up, but returned items do not increase Quantity On Hand.

Rule 2: I must always be allowed to receive less than the outstanding amount

It is very normal that I do not receive everything on the PO in one delivery. Maybe the vendor only shipped half, or maybe the vendor marked the rest as backordered.

The system must let me enter any amount from 0 up to the outstanding quantity for each item. If I receive zero today on a line, that item stays outstanding and the PO stays open.

For the returned column, the same rule applies — I enter the actual count of items I am sending back on that line. The system should make sure I am not returning items that do not exist in the shipment.

Rule 3: After receiving, the system must update QOH and QOO correctly

Once I confirm what I actually received, the system must update the inventory numbers:

  • QOH (Quantity On Hand) increases by the quantities I accepted
  • QOO (Quantity On Order) decreases by the same amount for this vendor’s PO only
  • Returned or damaged items do not affect QOH

This way, the shelf counts line up over time, and our reports do not show that we are still expecting stock that is already here or never coming.

Rule 4: The PO closes automatically when everything is fully received

If I have received everything on that PO and nothing is outstanding anymore, then my job on that order is done.

In that case:

  • The PO must automatically close (Closed = true)
  • The PO should no longer appear in the list of open POs for receiving

This is normal auto-close behavior and is different from forcing an order to close early.

5. Force Closing a Purchase Order

Sometimes the vendor tells us they cannot supply the rest of an order — maybe the vendor discontinued a part, or maybe the vendor is simply not going to ship it. In those cases, continuing to wait is not useful. That is when we use Force Close.

Force Close Rules

  • Only users with the right role (for example, a Shop Manager) should be able to use Force Close. (Store Staff are blocked from this action.)
  • The system must require a Force Close Reason before it allows the order to be closed this way. No reason means no force close.
  • Before the system actually force closes the order, the system must ask me one more time — “Are you sure you want to force close this PO?” — because once it is done, it cannot be undone. You cannot reopen a force-closed order, and we do not want anyone closing an order by accident.
  • When we force close a PO, the outstanding QOO must be removed for this vendor’s PO only. Other vendors’ POs for the same part must not be touched.
  • The PO must remain visible in history. The order is closed, not erased. We should still be able to see what happened.

Here is what the QOO cleanup looks like in practice: say we ordered 75 units of a part from Vendor A and over time we have received 42. If we force close, the system needs to reduce QOO by 33 for that part — that is the 75 we ordered minus the 42 we actually received. But if we also have an order for the same part from Vendor B, Vendor B’s QOO stays exactly where it was. We are only cleaning up this vendor’s numbers.

QOO decrease formula: Ordered - Total Previously Received (for each part on this PO)

If I have been working through the shipment and already entered some received and returned quantities when the decision comes to force close, the system should process all of that work first — the same as pressing Receive — and then close the order on top of it. I should not lose the receiving I have already done just because we are closing early.

This makes sure our reports stop showing that more stock is coming from this vendor, while still keeping a proper record of what was ordered and what actually arrived.

6. Real-World Exceptions the System Must Support

Not every item that shows up at the dock ends up on our shelves. The system has to handle that reality.

Damaged Items

  • I must be able to record the damaged quantity.
  • Damaged items must not go into inventory (QOH does not increase for damaged items).
  • Damaged items require a return reason, just like other returns.
  • Damaged items must appear in receiving history so we know what happened over time.

Backordered Items

  • Sometimes I receive nothing for a line because the item was backordered.
  • In that case, the outstanding quantity stays the same.
  • The PO stays open until we either receive the rest or force close the order.

Unordered Items (Incorrect Items)

Vendors will sometimes ship things we never ordered. When that happens, the system has to help me record these items properly, but those items must never enter inventory.

  • I must be able to record the incorrect (unordered) item.
  • I must be able to return the entire quantity and give the reason (“Incorrect item”).
  • These items must not affect the PO or QOH.

Since these unordered items are not in our Part table, the system must store them in a separate place (the UnorderedPurchaseItemCart table) with:

  • Description
  • Quantity
  • VendorPartNumber — the vendor’s part number, not ours

This way, we still have a record of what the vendor actually sent, but we do not count it as stock we carry.

When I come across an unordered item, I type in the three pieces of information — description, vendor part number, and quantity — and press Insert to add it to my unordered items list. I can see everything I have added so far in a grid, and if I made a mistake, I can press Delete on that row to remove it before I finalize.

These items only exist on my screen until I press Receive or Force Close. The items are not saved anywhere until that moment. If I realize I typed something wrong in the input fields before inserting, I can press Clear to reset the input fields and start that entry over.

All three fields — description, vendor part number, and quantity — must be filled in before the system lets me insert. I cannot leave any of them blank.

Since unordered items are always going back for the same reason — “Incorrect item” — the system can fill that reason in automatically. It is not a judgment call; it is a fact. The vendor sent something we never ordered, so it goes back.

7. Actions I Take When I Am Done

Once I have finished entering everything — received quantities, returned quantities, unordered items — I use a few key actions to commit my work or start over. These actions control when data is saved to the database and when data is discarded.

Receiving the Shipment

  • Processes the received quantities, returned quantities, and unordered items I have entered.
  • If all outstanding items are received (all Outstanding values become zero), the order is automatically closed.
  • Clears all entered data and updates the Order Details to reflect what was just received.
  • The system should not let me press Receive if I have not actually entered anything — no received quantities, no returned quantities, no unordered items. If everything is blank, there is nothing to process, and the system should tell me that instead of processing an empty session.

Forcing an Order Closed

  • Forces an order to close, even if there are still outstanding parts.
  • Requires a Reason for Closing before the order can be force closed.
  • Also processes any received quantities, returned quantities, and unordered items I have entered in this session — the same as Receive — before closing the order.

Reason for Closing

  • This field is required when using the Force Close button. Without a reason, the order cannot be force closed.

Starting Over (Reset)

  • Clears all changes I have made in this receiving session to the Order Details, including: received quantities, returned quantities, and any reason I have entered.
  • Also clears all changes in the unordered items section.

Before the Reset button actually clears anything, the system should warn me — “You have unsaved changes. Are you sure you want to reset?” — because I do not want to accidentally erase my data entry just because I pressed the wrong button.

In other words, Reset lets me clear all entries for this session if I realize I have made a mistake, before I commit anything by using the Receive or Force Close buttons.

8. What Happens After I Finish Receiving

When I am satisfied with what I have entered and I press the Receive button, the system’s job is to finalize everything properly and then clear the screen so I can move on to the next shipment.

After Saving (Receiving)

  • Inventory is updated based on the items I actually accepted.
  • A new Receive Order ID is created to represent this session (each session gets its own record).
  • The system never goes back and changes a previous receiving record. Every time I process a shipment, that becomes its own entry — a snapshot of what I received that day. If I receive more of the same items next week, that is a new entry on top of the old one. The history keeps growing, and nothing gets rewritten. This way, I can always trace back exactly what happened and when.
  • QOH and QOO are updated correctly based on what I accept. (QOH increases, QOO decreases — for this vendor’s PO only.)
  • The PO’s status is updated: still open if there is outstanding quantity, or closed if everything is received.
  • The screen is cleared so I do not accidentally keep editing the same receiving session again.

9. What the System Must Never Do

To be absolutely clear, there are certain things this system should never do:

  • Never assume the shipment matches the PO or packing slip.
  • Never auto-fill receiving quantities for me.
  • Never close a PO unless all quantities are received (except with an explicit force close and a reason).
  • Never allow me to receive more than the outstanding quantity.
  • Never add returned or damaged items into inventory.
  • Never adjust another vendor’s QOO when force closing this vendor’s PO.
  • Never hide real history using RemoveFromViewFlag.

10. Summary of Expectations

If the receiving module follows all of these rules correctly, then it works the way I work. The system helps me keep our inventory clean and our records honest, instead of adding more confusion.

Before Receiving

  • Show only vendors that have one open PO.
  • Show only POs that are placed (Order Date is set) and have not been closed yet (Closed = false).
  • Let me pick the active PO and record my Employee ID and the Receive Date.

During Receiving

  • Show full PO details: what was ordered, what was already received, and what is outstanding.
  • Let me enter the actual quantities received today.
  • Prevent over-receiving (Received cannot exceed Outstanding) and require clear reasons for any returned items.
  • Support damages, overshipments, unordered items, and backorders.
  • Update QOH (increases) and QOO (decreases, this vendor’s PO only) correctly based on what I accept.
  • Allow multiple receiving sessions for the same PO over time.

After Receiving

  • Update inventory and PO status based on what I received.
  • Auto-close the PO when everything is fully received (all Outstanding values become zero).
  • Allow force close only when there is a valid reason, a confirmation prompt, and the right role is using it.
  • Adjust only that vendor’s QOO when force closing — other vendors’ QOO values are not touched.
  • Clear the screen and let me move on to the next shipment.

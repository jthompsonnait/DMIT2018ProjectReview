Purchasing Module Requirements (Client Narrative)

Explained by Alex, an experienced Purchaser

Introduction

My name is Alex, and I have been a Purchaser for almost 20 years. I do not know how to program, but I know exactly what I need this purchasing system to do, because these are the decisions I make every day.

The purpose of this document is to explain what the system must accomplish, not how to build it. I will describe what I expect as the end user — the client — across the two major parts of the process:

  • The Vendor Selection step
  • The Purchase Order step

This is the same information I would give to a software team building this system for me.

1. Vendor Selection Screen Requirements

This is always the first step before creating or working on any order.

1.1 What I Need to See

I need a list of all active vendors with enough basic information so I can recognize them:

  • Company name
  • Their full address — street, city, province, and postal code
  • Phone number

A simple list is enough — I just need to know who I am selecting.

1.2 What I Need to Do

I need to:

  • Select a vendor
  • Confirm the selection

Once I choose a vendor, the system must decide what to show me next.

1.3 What Should Happen After I Select a Vendor

There are two, and only two, correct paths:

A. If an unplaced order already exists for that vendor

The system must:

  • Load that order exactly as I left it
  • Not generate suggestions or rebuild anything
  • Bring me back to the existing unplaced order with all its items

This is because I have already made decisions for that vendor.

B. If no unplaced order exists for that vendor

The system must:

  • Create a brand-new suggested order
  • Use inventory levels to determine which items might be needed

Suggestions must ONLY happen when starting a new order — never when one already exists.

1.4 One Order Per Vendor

We only work on one order per vendor at a time. We are a smaller company, and managing cash flow means I never need two draft orders for the same supplier. If an unplaced order already exists for a vendor, that is the order I work on — there is no reason to start a second.

2. Purchase Order Screen Requirements

After I select a vendor, I arrive at the main working screen — where I do most of my purchasing work. I cannot tell you how it should look, but I can tell you what it must do.

2.1 Purchase Order Information I Must See

The system must clearly show:

  • The vendor name
  • Their address and phone number
  • The PO number (or an indication that this is a new order)
  • Order totals: subtotal, GST, and final total

This helps me confirm I am building the order correctly.

3. Suggested Orders (Only for New Orders)

Suggested orders are only created when:

  • I selected a vendor
  • There is no existing unplaced order
  • I am starting a new order

The system identifies items that might need restocking using this rule:

Reorder Level > (Quantity On Hand + Quantity On Order)

Only items meeting that rule should appear in the suggested order.

The system should suggest ordering enough to bring us back up to the level we try to maintain. The suggested quantity should equal the difference between the reorder level and what we already have or have on the way. In other words:

Suggested Quantity = Reorder Level − Quantity On Hand − Quantity On Order

Suggestions Are NOT Decisions

A suggested order is only an initial suggestion. I might:

  • Change quantities
  • Remove items
  • Add new items the system did not suggest

The system must never save a suggested order automatically.

4. Working With Items in the Order

This is where the majority of my decision-making happens.

4.1 What I Need to Know for Each Item

For each part included in the purchase order, I need to see:

  • A way to identify exactly which part I am looking at
  • What the item is
  • How many we currently have
  • The level we should keep in stock
  • How many are already incoming
  • The price
  • The quantity I plan to order

I need to be able to change both the price and the quantity.

4.2 Totals Must Update Instantly

Any time I change:

  • A price
  • A quantity
  • Add a part
  • Remove a part

The totals must update immediately. Purchasers manage budgets continuously — we cannot wait until saving to see the financial impact.

4.3 Available Parts — What Else Can I Add?

Besides the items already in my order, I need to see what else this vendor sells that is not already in my order — a browsable list I can look through and pick from.

For each available part, I need to see:

  • The part identifier
  • What the item is
  • How many we currently have in stock
  • The level we try to maintain
  • How many are already on order
  • How far below the reorder level we are — so I can quickly see which items need reorder most
  • The price

I need a way to add an item from this list to my order with a single action.

How the Two Lists Work Together

A part should never appear in both lists at the same time. When I add something from the available list, the part moves into my order. When I remove something from my order, the part goes back to the available list. I should never see the same part listed twice in my order — the system should prevent that completely.

4.4 Price Adjustments

Prices are not always fixed. In real purchasing, I often need to change the price of an item for valid business reasons:

  • Volume discounts — A vendor offers a reduced price when I order a large quantity. For example, a part normally costs $12.00 each, but the vendor offers $10.50 each if I order 50 or more.
  • Promotional or discounted products — A vendor runs a temporary promotion or offers a discounted price on certain items.
  • Zero-price items — Sometimes a vendor includes items at no cost. For example, the vendor sends free marketing materials (catalogs, display stands), or the vendor compensates for a previous mistake or delay by including extra items at zero dollars.

The system must allow me to enter any price that is zero or greater (any non-negative value). A price of zero is valid when there is a business reason for it. The system should not block saving or placing an order because an item has a zero price.

5. Quantity Rules That Prevent Mistakes

These rules are extremely important in real purchasing work.

Rule 1: New Items Must Start at Quantity Zero

If I add an item from the general list of available parts, the default quantity must be zero.

This prevents accidental ordering. A quantity of zero forces me to consciously decide the quantity.

Rule 2: Suggested Items That Are Removed and Re-Added Must Also Start at Zero

If I remove a suggested item and later add it back, the quantity must not return to the original suggestion. The quantity must reset to zero.

Because adding something back means I am re-evaluating. I must make the quantity decision myself again.

Rule 3: Zero Quantity Blocks Saving

I cannot save or place an order if any item still has a quantity of zero. Zero is only the starting point — the system should remind me to enter a quantity of at least one before I commit. Every line in my order must have a quantity of at least one before I can save or place the order.

6. Saving and Updating an Order

I need to be able to save progress without placing the order.

  • If the order is new (no PO number yet) → I use Save
  • If the order is an existing unplaced order (already has a PO number) → I use Update

This allows me to return later and continue working where I left off.

Every item must have a price entered. The price can be zero when there is a valid business reason — for example, marketing materials or vendor compensation. The system should not block saving or placing an order because of a zero price.

7. Placing the Order

Placing the order is the point of commitment.

When I place an order, the system must:

  • Record the date the order was placed
  • Update the inventory so items are considered “on order” (this increases Quantity On Order for each part)
  • Lock the order so nothing can be edited
  • Disable delete
  • Disable update

A placed order cannot be changed.

8. Deleting Orders (Updated Logic)

Deleting an order depends on whether anything has been saved to the database yet.

Whenever I delete anything — the whole order or just a single line — the system should confirm with me first. Deleting is serious, and I do not want to do it by accident.

There are three important rules.

8.1 If the Order Has NOT Been Saved (No PO Number Yet)

If I am building a new order that has not been saved:

  • Deleting the order should simply discard the order
  • Nothing should be stored
  • No “hidden” flags should be used

This is unsaved work being discarded — nothing was ever recorded.

8.2 If a Line Item Has NOT Been Saved Yet

If I added a line item but have not saved the order since adding that line:

  • Deleting that line should simply remove it
  • Again, nothing should be saved or hidden

The line never officially existed.

8.3 If the Order OR a Line Item HAS Been Saved Before

If the order has a PO number, or a line has been saved in the past:

  • Deleting the order should not erase it — instead, the system should mark the order as removed from view (soft delete using RemoveFromViewFlag)
  • When I delete a whole order, I only need to mark the order itself as removed. The line items belong to that order, so the line items are included automatically
  • Deleting an individual saved line item should mark just that line as removed from view (soft delete using RemoveFromViewFlag)

This is the proper business process:

  • Unsaved drafts are discarded completely
  • Saved records use soft delete for auditing and history

8.4 Placed Orders Cannot Be Deleted

If an order has already been placed:

  • The delete button must be disabled

A placed order is an official communication with a vendor — it cannot be undone.

9. What the System Must Never Do

To be absolutely clear, there are certain things this system should never do:

  • Never auto-save a suggested order — suggestions are a starting point, not a commitment
  • Never apply suggestions to an existing order — if I already have a draft, leave it exactly as I left it
  • Never allow a placed order to be modified or deleted — once placed, it is locked
  • Never auto-fill quantities — new and re-added items always start at zero so I make conscious decisions
  • Never allow the same part to appear twice in one order — each part gets one line, period

10. Clearing My Work

If I want to reset and begin again or switch vendors:

  • The Clear command should reset everything
  • The Clear command should return me to the vendor selection
  • If I have unsaved changes, the system must warn me

This prevents lost work.

11. Summary of Expectations

Vendor Selection

  • Show only active vendors with full address details
  • Allow selecting a vendor
  • Load existing unplaced order OR create a suggested order
  • Only one open order per vendor at a time

Suggested Orders

  • Only for new orders
  • Based on inventory shortages
  • Suggested quantity = reorder level minus what is on hand and on order
  • Never applied to existing orders

Purchase Order Work

  • Show vendor info and PO info
  • Show each item with its part identifier
  • Show totals
  • Let me adjust prices and quantities
  • Totals must update instantly
  • Show available parts I can add, with urgency indicator
  • No duplicate parts in the same order

Quantity Rules

  • New items start at zero
  • Re-added suggested items start at zero
  • Every item must have quantity at least one before saving or placing
  • Every item must have a price entered (zero is allowed for valid business reasons such as marketing materials or vendor compensation)

Saving and Placing

  • Save → if new (no PO number yet)
  • Update → if existing (already has a PO number)
  • Place → locks the order and updates inventory

Deleting

  • Always confirm before deleting
  • Unsaved orders → discard completely
  • Unsaved line items → discard completely
  • Saved orders → soft delete the order (children follow the parent)
  • Saved line items → soft delete that line
  • Placed orders → cannot be deleted

Clear Behavior

  • Reset screen
  • Warn if unsaved changes exist

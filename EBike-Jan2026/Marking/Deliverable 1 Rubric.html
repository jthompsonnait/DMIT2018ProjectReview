NOTE: This is a high-level overview of the rubric. Please refer to the eBike project documentation and your instructor for any clarification.

Grading Criteria

1. Purchasing (15 marks)

# Criteria Marks Description
1 Vendor Selection and PO Loading 3 Users can select a vendor and either load an existing unplaced purchase order (exactly as last saved) or generate a new suggested order if none exists.
2 Suggested Order Generation 4 The system correctly suggests parts based on stock levels using the formula: ReorderLevel - (QOH + QOO) > 0. Suggestions are generated only for new orders, never applied to existing unplaced orders.
3 Parts Management (Two-Grid Interaction) 3 Users can add parts from the available parts grid, modify quantities and prices in the order grid, or remove parts (which return them to the available list). Newly added and re-added parts default to quantity zero. No duplicate parts allowed in one order.
4 Order Totals Calculation 2 The Subtotal, GST (5%), and Total dynamically update as parts are added, removed, or modified.
5 Business Rules and Validation 3 Quantity to Order must be at least 1, Price must be at least 0, no duplicate parts in one order, confirmation prompts for delete actions, and buttons are disabled when the form is not valid.
Total for Purchasing 15

2. Receiving (15 marks)

# Criteria Marks Description
1 Purchase Order Selection 3 Users can search for and select outstanding purchase orders (placed but not yet closed). The grid shows PO Number, Order Date, Vendor, and Contact Number.
2 Receiving and Returns Handling 4 Users can enter received quantities and return quantities for each line item. The grid displays all 7 columns: Part ID, Description, Quantity Ordered, Outstanding (Ordered minus Total Previously Received), Received, Returned, and Reason. Received quantities cannot exceed the Outstanding amount.
3 Return Validation Rules 3 The system requires a reason for every returned item. Valid reasons are: Overship, Incorrect item (unordered only), and Damaged on arrival. Return quantities cannot exceed what was actually received in the current shipment. Validation must be handled in the UI.
4 Unordered Items 3 Users can add items not on the purchase order using a three-field form (Description, Vendor Part Number, Quantity). The UI provides Insert (add to grid), Delete (remove from grid), and Clear (reset input fields) actions. All three fields are required before inserting.
5 Order Closure Logic 2 The Force Close option requires a reason to be entered and a confirmation prompt before executing. Force Close is only available to the Shop Manager (appropriate for D1 — the actual closure does not save to the database in Deliverable 1).
Total for Receiving 15

3. Servicing (15 marks)

# Criteria Marks Description
1 Customer and Vehicle Selection 3 Users can search for customers by last name, select a vehicle from a dropdown, and confirm the VIN before proceeding. UI validation must ensure valid lookups. Changing the customer clears any existing service order data (with a warning).
2 Service Entry and Service Cart 4 Users can add standard services (predefined description and hours from a dropdown) or custom services (user-entered description and hours). Hours must be greater than zero. The shop rate is always $65.50 per hour (non-editable). The service cart displays: Service Name, Hours, Rate, Extended Price (Hours multiplied by $65.50), and a Delete action.
3 Parts Selection and Parts Cart 4 Users select a category first, then browse available parts showing Part Name, Price, QOH, and Quantity to Add. Quantity must be at least 1 and must not exceed QOH. Duplicate parts increase the existing quantity (no duplicate rows). The parts cart displays: Part Name, Price, Quantity, Total, and a Delete action.
4 Totals and Coupon Calculation 2 The system dynamically calculates Labour Subtotal, Parts Subtotal, combined Subtotal, Discount (if a coupon is verified), GST (5% after discount), and Total. Coupon verification requires entering a code and clicking Verify — the discount is only applied after successful verification.
5 Validation and Registration Rules 2 Users cannot register a service order unless a customer, vehicle, and at least one service entry are present. Parts are optional. UI must validate all inputs before submission. Buttons are disabled when prerequisites are not met.
Total for Servicing 15

4. Sales and Returns (15 marks)

# Criteria Marks Description
1 Customer Search and Navigation 2 Users can search for customers by partial phone number and select a customer. Navigation buttons (Customer Search, Product Selector, View Cart, Checkout, Cancel) are always visible. Switching customers clears the cart and coupon data (with a warning).
2 Product Selection and Cart Management 4 Users select a category to view available products, enter a quantity (at least 1, must not exceed QOH), and add to the cart. Duplicate items increase the existing quantity (no duplicate rows). The cart displays: Part, Price, QOH (read-only), Quantity (adjustable), Extended Price, and Delete. Items with zero QOH, discontinued status, or no selling price cannot be added.
3 Checkout and Sale Completion 2 The checkout screen shows a final summary of all items, coupon discount (if verified), tax (5% after discount), and total. A payment type (Cash, Debit, or Credit) must be selected. The sale button is disabled until a customer is selected, at least one item is in the cart, and a payment type is chosen.
4 Returns Processing and Validation 5 Users enter a Sale Invoice number to load the original sale items. Each item shows: Part Name, Original Quantity, Previously Returned (from all past returns on that invoice), Price, Refundable status, and editable fields for Quantity to Return and Reason. Non-refundable items and fully returned items have the Quantity to Return field disabled. Maximum returnable quantity is Original Quantity minus Previously Returned. A reason is required for every returned item.
5 Order Totals and Discounts 2 For sales: Subtotal, coupon discount (percentage-based, applied before tax), GST (5%), and Total update dynamically. For returns: the refund calculation honours the original discount percentage (even if the coupon has since expired). Totals update in real time as quantities change.
Total for Sales and Returns 15

Additional Notes

UI/UX Considerations

Ensure that buttons, forms, and grids are clearly labelled, responsive, and provide user feedback.

Validation and Error Handling

Proper warnings and confirmations should be implemented for invalid inputs and irreversible actions (such as deleting items or clearing unsaved data).

Back-End Validation Not Required

Deliverable 1 focuses on retrieval and UI-based validation. Back-end data integrity is enforced only for data retrieval.

Data Flow and Live Updates

The system should dynamically update order totals and validation messages without requiring manual refreshes.

Do Not Save Data

The focus of Deliverable 1 is the retrieval of data and UI validation. A button that would save to the database should have validation done on the UI (for example, a reason is needed before Force Close in the Receiving module), but it does not save to the database.

Group Project

Although this project is mainly focused on individual modules within the project, you are expected to meet with your group and test each other’s work. Within large companies, one of your daily requirements is to do “smoke testing”.

Smoke testing is a broad but shallow testing process that verifies the core functions of a software build after a significant change or addition has been made. It aims to quickly identify major problems and ensure the build is stable enough for further testing.

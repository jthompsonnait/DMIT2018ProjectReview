Security (5 marks)

1. Authentication Redirect (1 mark)

Users who are not logged in are redirected to the login page. No module page should be accessible without authentication.

2. Unauthorized Role Warning (1 mark)

Users who are logged in but do not have the appropriate role for a page will see a warning message indicating they do not have permission to access that page.

3. Employee Identity Display (1 mark)

Users with valid roles will have their full name, role, and EmployeeID displayed on the page. This confirms to the user which account is active and what permissions they have.

4. Role-Based Feature Restriction (1 mark)

Specific features are disabled or hidden based on the user’s role. Each module has one restricted feature:

Module Restricted Feature Who Is Blocked
Purchasing Place Order button Store Staff cannot place orders
Receiving Force Close button Store Staff cannot force close
Servicing Coupon area Mechanic cannot access the coupon area
Sales and Returns Returns section Salesperson cannot access returns

The restricted elements (buttons, text boxes, or entire sections) must be disabled or hidden — not just visually greyed out with no enforcement.

5. Manager Full Access (1 mark)

Manager roles have full access to all features and pages within their module. No features are restricted for the manager. The Admin account (a code-level super-role, not a database role) also has full access to everything.

Save Function (7 marks)

1. Business Rule Validations (2 marks)

All business rules are validated before saving, using the BYSResults pattern (methods return lists of success and error messages). Both front-end (UI) and back-end (BLL) validations must be in place. The UI must display all error messages returned by the business logic.

Examples per module: - Purchasing: QTO must be at least 1, Price must be at least 0, no duplicate parts - Receiving: Received quantity must not exceed Outstanding, reason required for returns, all three fields required for unordered items - Servicing: At least one service entry required, hours greater than zero, part quantity must not exceed QOH - Sales and Returns: Quantity must not exceed QOH (sales), return quantity must not exceed purchasable minus previously returned (returns), reason required for each return

2. Entity Updates — Parent-Child (2 marks)

The save function correctly creates and updates both parent and child entities in the proper order within a single transaction.

Parent-child relationships per module: - Purchasing: PurchaseOrder (parent) → PurchaseOrderDetail (children); on Place, also updates Part.QOO - Receiving: ReceiveOrder (parent) → ReceiveOrderDetail + ReturnedOrderDetail + UnorderedPurchaseItemCart (children); also updates Part.QOH and Part.QOO - Servicing: Job (parent) → JobDetail (services) + JobDetailPart (parts) (children); also updates Part.QOH - Sales and Returns: Sale (parent) → SaleDetail (children); also updates Part.QOH. Returns: SaleRefund (parent) → SaleRefundDetail (children); also updates Part.QOH

3. Error Handling (1 mark)

The application uses appropriate exception types and exception properties to handle errors during the save process. Errors are caught, logged or displayed to the user, and do not crash the application. The user receives clear feedback when something goes wrong.

4. Data Persistence (2 marks)

Data is correctly saved to the database using OLTP (transactional) methods to maintain data integrity. All related changes happen within a single transaction (if any part fails, nothing is saved).

Key persistence actions: - Inventory adjustments: QOH increases (receiving, returns) or decreases (sales, servicing); QOO increases (placing PO) or decreases (receiving) - Post-save UI state: forms are locked or reset as appropriate (for example, a placed purchase order becomes read-only, a completed sale disables all buttons) - Generated IDs (PO number, Job ID, Sale ID) are displayed to the user after successful save

Penalties (Up to -5 marks)

A penalty of up to 5 marks can be deducted for the following:

  1. Missing validation for methods with parameters — for example, a BLL method that accepts an ID but does not check whether it is valid before proceeding
  2. Weak or incorrect integer validation — not following programming standards for checking numeric input
  3. Not using appropriate exception types or exception properties — for example, using generic Exception instead of ArgumentException or ArgumentNullException where appropriate
  4. Other violations of general programming practices — such as not following the patterns taught in class, not using secrets for connection strings, or missing code comments

Project Failure to Run

Due to the time that has been allocated to you for the project, any individual module that fails to run successfully will earn zero marks for that individual. Please work with both your group and your instructor to address any questions you may have.
